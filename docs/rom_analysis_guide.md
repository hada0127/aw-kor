# Game Wars GBA ROM 분석 가이드

이 가이드는 Game Wars GBA ROM을 분석하는 단계별 방법을 설명합니다.

## 필수 도구

- **VisualBoyAdvance M**: ROM 실행 및 메모리 디버깅
- **HxD Hex Editor**: 바이너리 파일 분석
- **Crystal Tile 2.5**: 그래픽 및 텍스트 분석

## 1단계: ROM 기본 정보 읽기

### 1.1 ROM 헤더 확인

HxD에서 ROM을 열고, 오프셋 `0xA0` 부터 확인:

```
0xA0 - 0xAB: 게임 제목 (12바이트, ASCII)
0xAC - 0xAF: 게임 코드 (4바이트, ASCII)
0xB0 - 0xB1: 제작사 코드 (2바이트, ASCII)
0xB2:        고정값 0x96
0xB3:        메인 단위 유형
0xB4 - 0xB7: 장치 크기
0xB8 - 0xBB: 예약
0xBC:        버전 번호
0xBD:        체크섬 (0-255)
0xBE - 0xBF: 예약
```

Game Wars 예상 정보:
- 제목: "GAME WARS" 또는 일본어 제목
- 게임 코드: 미정 (분석 필요)

### 1.2 ROM 파일 크기 확인

일반적인 GBA ROM 크기:
- 16MB (0x1000000 바이트)
- 32MB (0x2000000 바이트)

## 2단계: 게임 플레이 및 스토리 이해

### 2.1 VisualBoyAdvance M에서 ROM 실행

1. VisualBoyAdvance M 실행
2. File → Open ROM → `original/game_wars_1.gba` 선택
3. 게임이 제대로 실행되는지 확인
4. 메뉴, UI, 텍스트 등을 다양하게 테스트

### 2.2 텍스트 위치 기록

게임을 플레이하면서 발견한 텍스트를 기록:

| 텍스트 | 화면 위치 | 예상 주소 | 비고 |
|--------|---------|---------|------|
| メニュー | 메인 화면 | ? | 메뉴 텍스트 |
| ゲーム開始 | 메인 메뉴 | ? | 게임 시작 |
| 新規ゲーム | 모드 선택 | ? | 새 게임 |

## 3단계: 메모리 분석

### 3.1 VisualBoyAdvance M 디버거 사용

1. Debugger 메뉴 열기
2. Memory Viewer 탭에서 메모리 검색
3. 게임 플레이 중 특정 텍스트가 메모리의 어디에 위치하는지 추적

### 3.2 문자 인코딩 파악

Game Wars는 아마도 다음 중 하나를 사용:
- **Shift-JIS**: 일본 게임에서 가장 흔함
- **EUC-JP**: 때로 사용됨
- **Custom Encoding**: 게임 개발사가 정의한 독자 인코딩

### 3.3 텍스트 데이터 블록 찾기

HxD에서 ROM을 열고, 알려진 일본어 텍스트를 검색:

```
Ctrl+H (Find/Replace)
"メニュー" 또는 이와 유사한 일본어 텍스트 입력
```

발견된 위치를 기록하면 텍스트 데이터 블록의 범위를 파악할 수 있습니다.

## 4단계: .tbl 파일 생성

### 4.1 기본 구조 이해

`.tbl` (Table) 파일은 바이트와 문자 간의 맵핑을 정의합니다:

```
00=あ      # 바이트 0x00은 문자 'あ'
01=い      # 바이트 0x01은 문자 'い'
20=        # 바이트 0x20은 공백
FF=\end    # 특수 표시
```

### 4.2 HxD에서 .tbl 파일 로드

1. HxD 설정에서 .tbl 파일 로드
2. ROM을 열면 바이트가 자동으로 문자로 변환되어 표시
3. 이를 통해 어떤 바이트가 어떤 문자에 대응되는지 확인

### 4.3 초안 .tbl 파일 작성

ROM 분석을 통해 발견한 바이트-문자 맵핑을 `tools/game_wars.tbl`에 작성:

```
# Game Wars 바이트 맵핑
00=あ
01=い
02=う
...
# 주의: 정확한 바이트값은 게임 분석으로만 결정 가능
```

### 4.4 검증

생성한 .tbl 파일로 ROM을 다시 보면서:
- 텍스트가 올바르게 표시되는지 확인
- 누락되거나 잘못된 맵핑 수정
- 반복 조정

## 5단계: 포인터 테이블 작성

### 5.1 포인터 개념

포인터는 ROM의 특정 위치에 저장된 다른 위치의 주소입니다.
일반적으로 4바이트 (32비트, little-endian):

```
주소 0x100010에 포인터 있음: 0x0A 0x00 0x20 0x00
→ little-endian으로 읽으면: 0x00200A (텍스트 주소)
```

### 5.2 포인터 위치 찾기

1. HxD에서 이미 발견한 텍스트 주소를 알고 있음
2. 그 주소를 역으로 검색하여 포인터를 찾기:
   - 예: 텍스트가 0x200000에 있으면
   - 0x00 0x00 0x20 0x00 (little-endian) 검색
3. 발견된 위치가 포인터 테이블

### 5.3 포인터 테이블 CSV 생성

`data/pointers.csv`:

```csv
pointer_address,text_address,size
0x100010,0x200000,4
0x100014,0x200010,4
0x100018,0x200020,4
```

## 문제 해결

### Q: ROM에서 일본어 텍스트를 찾을 수 없습니다.

**A**: 
- ROM 파일이 손상되었을 수 있음 (체크섬 확인)
- 텍스트가 압축되어 있을 수 있음 (특별한 압축 해제 필요)
- 커스텀 인코딩을 사용할 수 있음 (더 깊은 분석 필요)

### Q: .tbl 파일을 만들었는데 텍스트가 이상하게 표시됩니다.

**A**:
- 인코딩 확인 (Shift-JIS인지 다른 것인지)
- 바이트 값이 잘못되었을 수 있음 (재분석)
- 멀티바이트 문자 처리 확인 (일본어 문자는 2-3바이트일 수 있음)

### Q: 포인터를 찾을 수 없습니다.

**A**:
- little-endian vs big-endian 확인 (GBA는 little-endian)
- 포인터가 압축되어 있을 수 있음
- 동적으로 계산되는 포인터일 수 있음 (메모리에서만 존재)
